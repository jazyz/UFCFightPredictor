## Production Training

The deployable model is trained by:

```bash
.venv/bin/python train_final_model.py \
  --train-through 2026-06-27 \
  --params data/best_params.json
```

Training flow:

- Load features from `data/detailed_fights.csv`.
- Keep rows from `2009-01-01` through the requested `--train-through` date.
- Drop ID columns like fighter names, title, and date.
- Select usable numeric features and remove highly correlated columns above `0.95`.
- Encode `Result` as binary `win` / `loss`.
- Duplicate training rows with red and blue fighters swapped, then invert the target.
- Fit a LightGBM `LGBMClassifier`.
- Save model artifacts into `saved_models/` and `saved_preprocessing/`.

Current saved model metadata:

- `train_through`: `2026-06-27`
- `training_rows`: `3992`
- `feature_columns`: `164`
- `dropped_correlated_columns`: `28`
- model: `saved_models/lgbm_single_model.joblib`
- metadata: `saved_models/lgbm_single_model_metadata.json`

## Feature Engineering

`process_fights_alpha.py` rebuilds `data/detailed_fights.csv` chronologically.

For each fight, it creates features using only each fighter's prior history, then updates fighter stats after that fight. Features include ELO, age, layoffs, streaks, wins, title wins, opponent ELO, weighted fight stats, differentials, and `oppdiff` matchup columns.

`oppdiff` means:

```text
feature oppdiff = Red feature - Blue feature
```

Example:

```text
Red age = 30
Blue age = 34
age oppdiff = -4
```

## Leakage-Safe Backtesting

The honest evaluator is:

```bash
.venv/bin/python testing/no_leakage_backtest.py \
  --start-date 2025-06-27 \
  --end-date 2026-06-27 \
  --output-dir test_results
```

For every event date, it:

- trains only on fights with `Date < event_date`
- performs feature selection only on that event's training rows
- fits a fresh temporary LightGBM model
- predicts that event's fights
- uses odds and winners only after prediction to score bets and bankroll

This is why the one-year run fit `42` models: there were `42` event/card dates in the evaluation window. The two-year run fit `84` models.

By default, this backtester uses built-in LightGBM params instead of `data/best_params.json`, because tuned params may have been selected using future data.

## Betting Strategy

The current default safe-backtest strategy is:

```bash
--strategy 0.05,0.05,0.005
```

The three values mean:

- `0.05`: bet 5% of Kelly
- `0.05`: cap any bet at 5% of bankroll
- `0.005`: use a 0.5% bankroll flat fallback/floor

With a `$1000` bankroll:

- positive Kelly edge: bet `bankroll * 0.05 * kelly`, capped at `$50`, floored at `$5`
- slightly negative Kelly but not terrible: bet `$5`
- very negative Kelly: no bet
